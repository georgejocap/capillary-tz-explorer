Three timezone concepts
Org Timezone
🏢
Your company's clock
Set in Organization Settings — usually HQ location. Most UI screens display dates in this timezone.
Asia/Kolkata · Asia/Singapore · UTC
Server Timezone
🖥️
Where your data lives
The timezone of the Capillary cluster hosting your org. All API responses return timestamps in this timezone.
India +05:30 · EU UTC · APAC2 +08:00
Input Timezone
📍
Where the action happened
When data arrives with an offset (e.g. purchase from Dubai), Capillary preserves it for local-time promotion evaluation.
billingTimeInputOffset · eventDateTimeInputOffset

API Behavior
How date and time values work in Capillary APIs.
Plain English: Every date in a GET response is in the server's timezone — not yours. India cluster = IST (+05:30), EU = UTC. All dates are ISO 8601.
EU Cluster (UTC)
{ "expiryDate": "2025-12-16", ← old field (deprecated) "expiryDateISO": "2025-12-16T14:30:45Z" ← use this }
India Cluster (IST) — same moment
{ "expiryDateISO": "2025-12-16T20:00:45+05:30" }
Tip: Always use the *ISO fields — old fields are being phased out and contain no timezone info.
Plain English: Send dates in ISO 8601 with your timezone offset. For modules that support it, also pass timeZoneName.
Creating a Promotion
{ "startOnISO": "2026-03-01T00:00:00+05:30", "endOnISO": "2026-03-31T23:59:59+05:30", "timeZoneName": "Asia/Kolkata" }
Legacy vs preferred
{ "startOn": "2026-03-01 00:00:00", ← no TZ info "startOnISO": "2026-03-01T00:00:00+05:30" ← preferred }
Plain English: Transactions return in server timezone plus the original offset — letting you reconstruct the customer's local time for promotion evaluation.
Send a transaction
POST /transactions { "billingDate": "2026-01-23T09:00:00+05:30" }
Fetch it back (EU cluster)
{ "billingTime": "2026-01-23T03:30:00Z", ← server TZ "billingTimeInputOffset": "+05:30" ← original kept } // Reconstruct local: 03:30 UTC + 05:30 = 09:00 local ✓
Why it matters: Without the offset, a "9–11 AM" promotion evaluates against 03:30 UTC and misses. The offset field enables correct local-time evaluation.
Plain English: Behavioural events are stored as epoch milliseconds — timezone-neutral. The original offset is stored alongside for campaign window evaluation in the customer's local time.
Send an event
{ "eventDateTime": "2024-08-24T21:45:00+05:30" }
Stored and returned as
{ "eventDateTime": 1724516100000, ← epoch ms "eventDateTimeInputOffset": "+05:30" ← offset kept }

Important: All Rewards Catalog API inputs must be in UTC. All GET responses also return UTC. The legacy Marvel UI has no timezone changes yet.
Setting a reward for 10 AM Dubai (UTC+4)
{ "startTime": "2025-04-25T06:00:00Z" ← 10:00 Dubai = 06:00 UTC }

What are shadow fields? Old date fields had no timezone info (e.g. expiryDate: "2025-12-16"). Capillary adds a new *ISO version alongside. Migrate to ISO fields — old ones are deprecated.

Setup Guide
What needs configuring, what's already on, and how to enable what isn't.
🆕 Orgs created from 2026
Everything is on by default. UI timezone display, API formats, loyalty processing — all work out of the box.
🏛️ Orgs created before 2026
ISO API formats ✓ · Shadow fields ✓ · UI display — support ticket needed · Loyalty processing — support ticket needed
Set Org Timezone in InTouch
1
Click the Profile icon (top right) → Organization Settings
2
Go to Organization Profile → click Edit
3
Click the "Set Org TimeZone" tab
4
In Time Zone, add all timezones your org operates in. These appear as options in Promotions, MemberCare, etc.
5
In Base Time Zone, select your HQ timezone — the default across all UI modules.
6
Click Submit. All UI modules will now show dates in your base timezone.
