📍 Capillary Platform · Timezone Reference

Understanding time
across every module

Dates and times behave differently depending on the module, server region, and org settings. This guide helps developers, PMs, and support teams understand exactly what timezone their data is in — and why.

The Three Timezone Concepts
Org Timezone
🏢
Your company's clock
Set in Organization Settings — usually your HQ location. Most UI screens display dates in this timezone.
e.g. Org = Singapore → all listing pages show "23 Jan, 9:00 AM SGT"
Server Timezone
🖥️
Where your data lives
The timezone of the Capillary server cluster hosting your org. All API responses return timestamps in this timezone.
India cluster = +05:30 · EU cluster = UTC · APAC2 = +08:00
Input Timezone
📍
Where the action happened
When data arrives with an offset (e.g. a purchase from Dubai), Capillary preserves it for local-time promotion evaluation.
9 AM Dubai (+04:00) stored separately → "morning purchase" promotion fires correctly

API Behavior
How date and time values work in Capillary APIs — formats, offset fields, and what to expect.
Plain English: Every date in a GET API response is in the server's timezone — not yours. India cluster = IST (+05:30), EU = UTC. All dates follow ISO 8601.
EU Cluster (UTC) — GET response
// Points expiry — EU cluster { "expiryDate": "2025-12-16", ← old field (will be removed) "expiryDateISO": "2025-12-16T14:30:45Z" ← new ISO field (use this) }
India Cluster (IST) — same data, different cluster
{ "expiryDateISO": "2025-12-16T20:00:45+05:30" ← same moment, IST offset }
Note: Response timezone always matches the server. Always look for the *ISO version of date fields — old ones are being phased out.
Plain English: Send dates in ISO 8601 format with your timezone offset. For modules that support it, pass a timeZoneName like Asia/Kolkata.
Creating a Promotion with timezone
{ "startOnISO": "2026-03-01T00:00:00+05:30", ← ISO format with offset "endOnISO": "2026-03-31T23:59:59+05:30", "timeZoneName": "Asia/Kolkata" ← informational }
Legacy vs preferred fields
{ "startOn": "2026-03-01 00:00:00", ← legacy (no TZ info) "startOnISO": "2026-03-01T00:00:00+05:30" ← preferred }
Plain English: Transactions come back in server timezone plus the original offset. This lets you reconstruct the customer's actual local time for promotion evaluation.
Sending a transaction from India
POST /transactions { "billingDate": "2026-01-23T09:00:00+05:30" }
Fetching it back (EU cluster = UTC)
{ "billingTime": "2026-01-23T03:30:00Z", ← converted to server TZ "billingTimeInputOffset": "+05:30" ← original offset kept } // Reconstruct: 03:30 UTC + 05:30 = 09:00 local ✓
Why this matters: Without the offset, a "9–11 AM" promotion evaluates against 03:30 UTC and misses entirely. The offset field lets you reconstruct the correct local time.
Plain English: Behavioural events are stored as epoch milliseconds — timezone-neutral. The original offset is kept alongside for campaign window evaluation in the customer's local time.
Sending an event
{ "eventDateTime": "2024-08-24T21:45:00+05:30" }
How it's stored and returned
{ "eventDateTime": 1724516100000, ← epoch ms (timezone-neutral) "eventDateTimeInputOffset": "+05:30" ← original offset kept }

Set Org Timezone in InTouch
1
Click the Profile icon (top right) → Organization Settings
2
Go to Organization Profile → click Edit
3
Click the "Set Org TimeZone" tab
4
In Time Zone, add all timezones your org operates in. These appear as options in Promotions, MemberCare, etc.
5
In Base Time Zone, select your HQ timezone. This is the default across all UI modules.
6
Click Submit. All UI modules will now show dates in your base timezone.
