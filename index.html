ℹ️
Organizations created from 2026 onwards have timezone standardization enabled by default. Older organizations must request enablement via a Capillary Product Support ticket.
🏢 Org Timezone
The headquarters timezone configured for your organization. Used for displaying dates in the UI across most modules. Acts as the scheduling anchor for promotions, campaigns, and journeys.
🖥️ Server Timezone
The timezone of Capillary's server cluster (India = +05:30, EU = UTC). Used for processing and storing all events and transactions. All GET API responses return timestamps in this timezone.
📍 Input Timezone
The timezone offset included in an API request or event payload (e.g., +05:30). When provided, Capillary preserves this separately so downstream systems can reconstruct the original local time.

🔄 The Problem Timezone Standardization Solves
❌ Before Standardization
• A reward set for 10:00 AM Dubai time (UTC+4) might appear as 11:30 AM IST in the India cluster — with no timezone label.

• Interaction timestamps in MemberCare showed cluster time, confusing support teams in different regions.

• A transaction created at 09:00 local time evaluated against a promotion window using 03:30 server time — missing the window entirely.

• No consistent way to tell what timezone a date was in.
✅ After Standardization
• All GET API responses use ISO 8601 with server timezone — always predictable.

• Input timezone offsets are preserved separately (billingTimeInputOffset, eventDateTimeInputOffset).

• UI modules display consistently in org timezone.

• Promotions can specify their own IANA timezone for accurate multi-region scheduling.

• Shadow ISO fields (*ISO) added alongside legacy fields for backward compatibility.
📐 ISO 8601 Format Reference
// EU Cluster (UTC) "billingTime": "2026-01-23T03:30:45Z" "billingTimeInputOffset": "+05:30" // India Cluster (IST) "billingTime": "2026-01-23T09:00:45+05:30" "billingTimeInputOffset": "+05:30" // Shadow ISO field pattern (backward compat) "expiryDate": "2025-12-16" // deprecated (legacy) "expiryDateISO": "2025-12-16T14:30:45Z" // new ISO field
⚠️ IANA Timezone Format
Used in Promotions and org settings:
Asia/Kolkata Europe/London America/New_York Asia/Shanghai Handles DST automatically.
⏱️ Epoch Milliseconds
Used in Behavioural Events for timezone-neutral ordering:
eventDateTime: 1724488200000 = milliseconds since Unix epoch. Never ambiguous — always UTC-based internally.
🔢 Offset Notation
UTC offset format used in inputs and preserved fields:
+05:30 (IST) +07:00 (WIB) +00:00 (UTC/Z) -05:00 (EST)

🔌 API Timezone Behavior
How date-time values are handled in Capillary's REST APIs (v2 and above). v1 APIs are excluded from standardization.
ℹ️
All GET API date-time fields are returned in ISO 8601 format using the server timezone. For older fields, shadow *ISO fields are added.
GET Response Pattern
// EU Cluster example (UTC) { "expiryDate": "2025-12-16", // Legacy — will be deprecated "expiryDateISO": "2025-12-16T14:30:45Z", // ISO shadow field (new) "enrollmentDate": "2025-12-16T14:30:00.000Z", // Legacy — will be deprecated "enrollmentDateISO": "2025-12-16T14:30:45Z" // ISO shadow field (new) } // India Cluster example (IST) { "expiryDateISO": "2025-12-16T14:30:45+05:30" // Server TZ = +05:30 }
Key Principle
The response timezone always matches the server timezone, regardless of any timezone offset sent in the request. Fields inside extendedFields are returned as stored — no conversion applied.
ℹ️
POST/PUT APIs accept date-time in ISO 8601 format. For entities with IANA timezone support, a timeZoneName field records the creation timezone (informational only — does not affect processing).
POST/PUT Pattern
// Creating a Promotion with IANA timezone { "startOnISO": "2026-03-01T00:00:00+05:30", // ISO shadow field for POST "endOnISO": "2026-03-31T23:59:59+05:30", "timeZoneName": "Asia/Kolkata" // Informational: stored + returned in GET // Does NOT affect processing logic } // Creating a Badge (ISO shadow field) { "startOn": "2026-03-01 00:00:00", // Legacy field (still works) "startOnISO": "2026-03-01T00:00:00+05:30" // New ISO field (preferred) }
✅
Transaction APIs preserve the original input timezone offset alongside the server-converted timestamp. This is the core mechanism enabling local-time reward eligibility.
Transaction GET Response
// Input (from POS in India) POST /transactions { "billingDate": "2026-01-23T09:00:00+05:30" } // GET Response (when server is EU cluster = UTC) { "billingTime": "2026-01-23T03:30:00Z", // Converted to server TZ "billingTimeInputOffset": "+05:30" // Original offset preserved } // Reconstruct local time: // 03:30Z + 05:30 = 09:00 local — correct for promotion window eval
Configuration: Disable Auto Conversion (V2 only)
Config flag: CONF_ORG_DISABLE_MACHINE_TIME_CONV
When enabled: billing time is stored exactly as provided (no UTC conversion). Only for v2 Transaction APIs. Must be assessed for downstream impact before enabling.
// With CONF_ORG_DISABLE_MACHINE_TIME_CONV = true // Input: 2026-01-23T09:00:00+05:30 "billingTime": "2026-01-23 09:00:00" // Stored as-is
⚠️
Rewards Catalog (Marvel) has its own timezone config flag. By default it uses server timezone. The legacy Marvel UI has no timezone changes yet.

⚡ Behavioural Events Timezone
How event timestamps are captured, stored, and preserved for downstream local-time evaluation.
eventDateTime
The event time stored as epoch milliseconds. Timezone-neutral — enables global ordering of events independently of local timezones. Always UTC-anchored internally.
eventDateTimeInputOffset
The original timezone offset from the input, preserved separately. Allows downstream systems to reconstruct the customer's local time for eligibility evaluation.
Input Scenarios
Input Provided Example eventDateTimeInputOffset What Capillary Does
Date-time with offset 2024-08-24T15:30:00+07:00 +07:00 Extracts offset, stores epoch, processes normally
Date-time without TZ 2024-08-24T15:30:00 Not stored Processes normally, no offset stored
UTC notation (Z) 2024-08-24T15:30:00Z +00:00 Normalizes to +00:00, stores offset
Missing / null null Not stored Processes without errors, no offset
Real-World Example
// Scenario: 9PM–10PM local time cart campaign, server = UTC // Customer adds to cart at 9:45 PM IST POST /events { "eventDateTime": "2024-08-24T21:45:00+05:30" } // Processed event stored { "eventDateTime": 1724516100000, // Epoch ms (UTC-anchored) "eventDateTimeInputOffset": "+05:30" // Original offset preserved } // Campaign engine evaluation: // 1724516100000 → UTC 16:15 | + 05:30 → LOCAL 21:45 ✓ // 21:45 is within 21:00–22:00 → REWARD TRIGGERED ✓ // Without offset: evaluates as 16:15 UTC → outside window → MISS

Setting Up Org Timezone in InTouch
1. Access Organization Settings
Click the Profile icon → Organization Settings → Organization Profile
2. Navigate to Set Org TimeZone Tab
This tab controls both the base timezone and supported timezones for multi-region orgs.
3. Configure Supported Timezones
Add all timezones your org operates in. These populate the timezone dropdowns in Promotions and MemberCare.
4. Set Base Timezone
Select the primary HQ timezone as the base. This is the default display timezone across all UI modules.
5. Submit
All UI modules will now display dates in your configured org timezone (when standardization is enabled).
