Understanding how time works
across every Capillary module

Dates and times can behave differently depending on the module, the server region, and your org settings. This guide helps everyone — developers, PMs, and support teams — understand exactly what timezone their data is in and why.

🏢
Organization Timezone
Your company's clock
The timezone set in your organization's settings — usually your HQ location. This is what most UI screens display dates in.
Example: If your org is set to IST (India), all listing pages show dates like "Jan 23, 2026 9:00 AM IST"
🖥️
Server Timezone
Where your data lives
The timezone of the Capillary server cluster hosting your org (India = +05:30, EU = UTC). All API responses return timestamps in this timezone.
Example: India cluster returns 2026-01-23T09:00:00+05:30. EU cluster returns 2026-01-23T03:30:00Z — same moment, different representation.
📍
Input Timezone
Where the action happened
When data comes in with a timezone (like a purchase from Dubai), Capillary preserves that offset so rewards can be evaluated in the customer's actual local time.
Example: A purchase at 9:00 AM Dubai (+04:00) is stored separately so a "morning purchase" promotion fires correctly.

API Timezone Behavior

How date and time values work in Capillary APIs — what format to send, what you get back, and the fields to look for.

Plain English: Every date you get back from a GET API is in the server's timezone (not yours). India cluster = IST (+05:30), EU cluster = UTC. All dates follow the ISO 8601 format so they're unambiguous.
EU Cluster (UTC) — GET response example
// Points expiry — EU cluster { "expiryDate": "2025-12-16", ← old field (will be removed) "expiryDateISO": "2025-12-16T14:30:45Z" ← new ISO field (use this) }
India Cluster (IST) — same data, different cluster
// Same point in time, returned in server TZ { "expiryDateISO": "2025-12-16T20:00:45+05:30" ← same moment, IST offset }
Note: The response timezone always matches the server, regardless of any timezone you sent in the request. Always look for the *ISO version of date fields — the old ones are being phased out.
Plain English: When creating things via API, send dates in ISO 8601 format with a timezone offset. For modules that support it, you can also pass a timeZoneName (like Asia/Kolkata) to record which timezone the entity was created in.
Creating a Promotion with timezone
{ "startOnISO": "2026-03-01T00:00:00+05:30", ← ISO format with offset "endOnISO": "2026-03-31T23:59:59+05:30", "timeZoneName": "Asia/Kolkata" ← for records; informational only }
Legacy fields (still work but prefer the ISO version)
{ "startOn": "2026-03-01 00:00:00", ← legacy (no TZ info) "startOnISO": "2026-03-01T00:00:00+05:30" ← preferred (has TZ info) }
Plain English: When you fetch a transaction, you get the time in the server's timezone plus the original offset it was created with. This lets you figure out the customer's actual local time — which is used for promotion eligibility.
Sending a transaction from India
POST /transactions { "billingDate": "2026-01-23T09:00:00+05:30" }
Fetching it back (when server is EU = UTC)
{ "billingTime": "2026-01-23T03:30:00Z", ← converted to server TZ "billingTimeInputOffset": "+05:30" ← original offset kept } // Reconstruct local time: 03:30 UTC + 05:30 = 09:00 local ✓ // Promotion window "9–11 AM local" → fires correctly
Why this matters: Without the original offset, a promotion window of "9–11 AM" would evaluate against 03:30 UTC and miss entirely. The offset field lets you reconstruct the correct local time.
Plain English: Behavioural events (add to cart, page view, etc.) are stored as epoch milliseconds — a universal number that never depends on timezone. The original timezone offset is stored alongside, so campaign windows can be evaluated in the customer's local time.
Sending an event
{ "eventDateTime": "2024-08-24T21:45:00+05:30" } // Customer added to cart at 9:45 PM IST
How it's stored and returned
{ "eventDateTime": 1724516100000, ← epoch ms (timezone-neutral) "eventDateTimeInputOffset": "+05:30" ← original offset kept } // Campaign: reward if add-to-cart between 9 PM – 10 PM local // epoch → UTC 16:15 + offset +05:30 → LOCAL 21:45 PM ✓ → reward fires

// Reward starts 10:00 AM Dubai time on April 25 // Dubai UTC+4 → 10:00 - 4h = 06:00 UTC { "startTime": "2025-04-25T06:00:00Z" ← always send in UTC } // GET response also returns: "2025-04-25T06:00:00Z"

What are shadow fields? Old date fields didn't include timezone info (e.g. expiryDate: "2025-12-16"). To avoid breaking existing integrations, Capillary adds a new *ISO version alongside. Migrate to the ISO fields — the old ones are being deprecated.

Setup Guide

What you need to configure, what's already on, and how to enable what isn't.

🆕 Organizations created from 2026
  • ✓Everything is on by default. UI timezone display, API formats, loyalty processing, promotion timezone selection — all work out of the box.
🏛️ Organizations created before 2026
  • ✓ISO API formats already available
  • ✓Shadow ISO fields available
  • ⚠UI timezone display — request via support ticket
  • ⚠Loyalty processing org TZ — request via support ticket
How to set your Org Timezone in InTouch
1
Click the Profile icon (top right) → Organization Settings
2
Go to Organization Profile → click Edit
3
Click the "Set Org TimeZone" tab
4
In Time Zone, add all the timezones your org operates in. These will appear as options in Promotions, MemberCare, etc.
5
In Base Time Zone, select your HQ timezone. This is the default shown across all UI modules.
6
Click Submit. All UI modules will now show dates in your base timezone.

What is not standardized
These modules use legacy timezone behavior and are not covered by timezone standardization:
🚫 Insights 🚫 Badges UI 🚫 Offers (UI & API) 🚫 Rewards Catalog / Marvel UI 🚫 v1 APIs (all) 🚫 extendedFields values (no conversion)
